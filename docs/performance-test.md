> 테스트 일자: 2026-04-19

## 1. 요금 조회 캐시 효과 테스트

**목적:** 동일 조건의 요금 조회 요청 반복 시 Redis 캐시가 응답시간을 얼마나 단축하는지 측정

**시나리오:**
- Warmup: VU 1명 × 5회 (캐시 미스 구간)
- Load: VU 50명 × 30초 (캐시 히트 구간)

**결과:**

| 지표 | 값 |
|------|----|
| 총 요청 수 | 13,302건 |
| 처리량 | 379 req/s |
| 평균 응답시간 | 10.98ms |
| 중앙값 (p50) | 4.71ms |
| p90 | 17.24ms |
| p95 | **24.08ms** ✅ (목표: 100ms 이하) |
| 최대 응답시간 | 889ms |
| 실패율 | **0.00%** |

**✅ 모든 임계값 통과**

**해석:**
- 캐시 히트 시 중앙값 4.71ms — DB 조회 없이 Redis에서 직접 반환
- p95 24ms로 목표(100ms)의 1/4 수준
- 초당 379건의 요금 조회를 처리하며 실패율 0%

## 2. 숙소 검색 부하 테스트

**목적:** 대규모 동시 검색 요청 처리 능력 측정 (필수 요구사항 3번)

**시나리오:**
- 0~10초: VU 0 → 10명 (램프업)
- 10~40초: VU 10 → 100명 (최대 부하)
- 40~50초: VU 100 → 0명 (램프다운)

**결과:**

| 지표 | 값 |
|------|----|
| 총 요청 수 | 4,358건 |
| 처리량 | 86 req/s |
| 평균 응답시간 | 5.93ms |
| 중앙값 (p50) | 3.98ms |
| p90 | 10.6ms |
| p95 | **14.1ms** ✅ (목표: 500ms 이하) |
| p99 | **40.8ms** ✅ (목표: 1,000ms 이하) |
| 최대 응답시간 | 346ms |
| 실패율 | **0.00%** |

**✅ 모든 임계값 통과**

**해석:**
- VU 100명 동시 요청에서도 p95 14ms — 검색 결과 캐싱 효과
- p99 40ms로 극단적인 요청에서도 안정적

## 3. 동시 예약 테스트

**목적:** 재고 N개인 객실에 다수 동시 요청 시 정확히 N건만 성공하는지 검증 (필수 요구사항 5번)

**시나리오:**
- 대상 객실: 스위트 킹 (room_type_id=2, 재고 4개)
- VU 20명이 동시에 동일 날짜 예약 시도
- 비관적 락 (`SELECT FOR UPDATE`)으로 동시성 제어

**결과:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
동시 예약 테스트 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
총 요청:    20건
예약 성공:  4건
예약 실패:  16건 (재고 없음)
예상 성공:  4건 (재고 수)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ PASS: 재고 초과 예약 없음
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**✅ 재고 초과 예약(오버부킹) 없음 확인**

**해석:**
- 20명 동시 요청 중 재고(4개)만큼 정확히 4건 성공
- 나머지 16건은 재고 없음으로 실패 처리
- 비관적 락으로 동시성 문제 완전 차단

## 성능 개선 포인트 (운영 환경 고려사항)

로컬 환경 기준으로 이미 충분한 수치이나, 실제 운영 환경에서는 아래 항목을 추가로 고려한다.

- **검색 캐시 키 전략 개선**: 현재 city + checkIn + checkOut + guestCount 조합. 도시 단위 캐시로 묶으면 캐시 히트율 향상 가능
- **DB 커넥션 풀 튜닝**: 동시 예약 증가 시 HikariCP 풀 크기 조정 필요
- **Redis Cluster**: 단일 Redis 장애 시 캐시 전체 무효화 방지를 위한 클러스터 구성
- **Read Replica**: 검색/요금 조회는 읽기 전용 복제본으로 분산 처리
